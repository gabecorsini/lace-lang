LSP & Editor Setup

The Lace Language Server implements the Language Server Protocol (LSP) and is bundled directly into the lace binary. Any LSP-capable editor can get rich Lace support with a few lines of configuration.

What the LSP provides

The Lace language server currently supports the following LSP features:

🔍 Hover

Hover over any symbol to see its inferred type and attached doc comment. Works for local variables, function parameters, stdlib functions, and user-defined functions.

✨ Completion

Context-aware autocompletion for all standard library namespaces and functions, as well as every user-defined symbol visible in the current scope. Trigger with your editor's usual completion key (often Ctrl+Space).

⚠️ Diagnostics

Type errors, undefined variables, and lint warnings are reported inline as you type. Diagnostics are re-run on every save (and on most editors, on every keystroke).

🗺 Go-to-definition

Jump directly to where a function or variable is defined — including across files in the same project. Use your editor's go to definition command (typically gd in Neovim, gd in Helix, F12 in VS Code).

🖌 Formatting

Opinionated, deterministic formatting for .lace files. The formatter respects a 2-space indent and enforces consistent spacing around operators. Trigger via your editor's format command or configure format-on-save.

Starting the server

The language server uses stdio transport — it reads LSP messages from stdin and writes responses to stdout. You never need to launch it manually; your editor spawns it automatically when you open a .lace file.

The command your editor must invoke is simply:

Shell
lace lsp

The lace binary must be discoverable on your system PATH. After installing via Cargo, confirm it is available:

Shell
$ lace --version
lace 0.1.0

The server detects the project root by looking for a lace.toml manifest or a .git directory, walking up from the file being edited. Single-file mode is also supported when no project root is found.

Neovim

The recommended setup uses nvim-lspconfig. Because Lace is not yet in the upstream lspconfig registry, you register the server manually.

Prerequisites

  • Neovim 0.8 or later
  • nvim-lspconfig installed
  • (Optional) nvim-cmp + cmp-nvim-lsp for full completion UI
  • lace binary on PATH

Configuration

Add the following to your init.lua or a dedicated lace.lua file sourced from your config:

init.lua / lace.lua
-- Neovim LSP configuration for the Lace programming language
-- Add this to your Neovim config (init.lua or a dedicated lace.lua file).
--
-- Prerequisites:
--   1. Install lace: cargo install --path crates/lace-cli (or add `lace` to PATH)
--   2. Install nvim-lspconfig: https://github.com/neovim/nvim-lspconfig
--
-- Usage: `lace lsp` is launched automatically when you open a *.lace file.

local lspconfig = require("lspconfig")
local configs = require("lspconfig.configs")

-- Register the lace-lsp server if not already registered
if not configs.lace_lsp then
  configs.lace_lsp = {
    default_config = {
      cmd = { "lace", "lsp" },
      filetypes = { "lace" },
      root_dir = lspconfig.util.root_pattern("lace.toml", ".git"),
      single_file_support = true,
      settings = {},
    },
    docs = {
      description = "Lace Language Server",
      default_config = {
        root_dir = [[root_pattern("lace.toml", ".git")]],
      },
    },
  }
end

lspconfig.lace_lsp.setup({
  on_attach = function(client, bufnr)
    -- Key mappings (adjust to your preferences)
    local opts = { noremap = true, silent = true, buffer = bufnr }
    vim.keymap.set("n", "gd", vim.lsp.buf.definition, opts)
    vim.keymap.set("n", "K",  vim.lsp.buf.hover, opts)
    vim.keymap.set("n", "<leader>f", vim.lsp.buf.format, opts)
    vim.keymap.set("n", "<leader>ca", vim.lsp.buf.code_action, opts)
  end,
  capabilities = require("cmp_nvim_lsp").default_capabilities(),  -- if using nvim-cmp
})

-- Register .lace file type (Neovim does not know about it by default)
vim.filetype.add({
  extension = {
    lace = "lace",
  },
})

Key bindings (defaults)

KeyAction
gdGo to definition
KShow hover (type + docs)
<leader>fFormat buffer
<leader>caCode actions

Helix

Helix has built-in LSP support — no plugins required. Add the Lace language definition and server configuration to your ~/.config/helix/languages.toml.

Prerequisites

  • Helix 23.05 or later
  • lace binary on PATH

Configuration

Merge the following into ~/.config/helix/languages.toml (create the file if it does not exist):

~/.config/helix/languages.toml
# Helix editor configuration for the Lace programming language
#
# Add this to your ~/.config/helix/languages.toml
# (or merge it into an existing languages.toml)
#
# Prerequisites:
#   - `lace` binary must be on PATH (cargo install --path crates/lace-cli)

[[language]]
name = "lace"
scope = "source.lace"
injection-regex = "lace"
file-types = ["lace"]
comment-token = "//"
indent = { tab-width = 2, unit = "  " }
language-servers = ["lace-lsp"]
roots = ["lace.toml"]

[language-server.lace-lsp]
command = "lace"
args = ["lsp"]

Key bindings (Helix defaults)

KeyAction
gdGo to definition
KShow hover documentation
Space fFormat buffer
Space aCode actions
Ctrl+SpaceTrigger completion

Zed

Zed supports extensions that bundle language configuration and LSP wiring. Install the Lace extension by dropping the following file into your Zed extensions folder.

Prerequisites

  • Zed 0.130 or later
  • lace binary on PATH

Installation

  1. Create the extension directory: ~/.config/zed/extensions/lace/
  2. Save the snippet below as extension.json inside that directory.
  3. Restart Zed. It will auto-detect the extension and start lace lsp when a *.lace file is opened.
~/.config/zed/extensions/lace/extension.json
// Zed editor extension snippet for the Lace programming language.
//
// To use:
//   1. Create a Zed extension directory: ~/.config/zed/extensions/lace/
//   2. Copy this file there as extension.json
//   3. Make sure `lace` binary is on PATH
//
// Zed will auto-detect the extension and start `lace lsp` when a *.lace file is opened.

{
  "name": "Lace",
  "version": "0.1.0",
  "description": "Lace programming language support for Zed",
  "authors": ["Gabe Corsini"],
  "repository": "https://github.com/gcorsini/lace",

  "languages": [
    {
      "name": "Lace",
      "grammar": "lace",
      "path_suffixes": ["lace"],
      "line_comments": ["// "],
      "block_comment": ["/*", "*/"],
      "brackets": [
        { "start": "{", "end": "}", "close": true, "newline": true },
        { "start": "[", "end": "]", "close": true, "newline": false },
        { "start": "(", "end": ")", "close": true, "newline": false }
      ],
      "indent_on_line_ends_with": ["{"],
      "decrease_indent_pattern": "^\\s*}"
    }
  ],

  "language_servers": [
    {
      "name": "lace-lsp",
      "language": "Lace",
      "command": "lace",
      "args": ["lsp"],
      "initialization_options": {}
    }
  ]
}

Troubleshooting

If the LSP does not start, work through the checklist below before filing a bug report.

⚡ lace binary not found / not on PATH

The most common issue. Verify the binary exists and is executable: 

$ which lace
/home/you/.cargo/bin/lace

$ lace --version
lace 0.1.0

If which lace prints nothing, either install lace (cargo install --path crates/lace-cli) or add its directory to PATH in your shell profile (~/.bashrc, ~/.zshrc, etc.): 

export PATH="$HOME/.cargo/bin:$PATH"

Note: many editors launch with a minimal environment and do not inherit your shell PATH. You may need to set PATH in your editor's own configuration or use the full absolute path to the binary (e.g. /home/you/.cargo/bin/lace) in the LSP command field.

⚡ lace lsp subcommand not found

This means you have an older lace binary installed that pre-dates the LSP feature. Reinstall from the latest source: 

cargo install --path crates/lace-cli --force

⚡ LSP attaches but no completions or diagnostics appear

Check that the filetype is being detected as lace. In Neovim run :set filetype?; it should print filetype=lace. If not, make sure the vim.filetype.add call from the config snippet is being executed.

In Helix, run :lang in command mode to confirm the active language is lace.

⚡ Root directory not detected; cross-file features missing

The server walks upward from the open file looking for lace.toml or .git. If neither exists, it falls back to single-file mode and cannot resolve imports or provide cross-file go-to-definition.

Create a lace.toml at the project root (it can be empty) to anchor the project: 

touch lace.toml

⚡ Viewing LSP logs

Neovim: open the log with :LspLog or inspect ~/.local/state/nvim/lsp.log.

Helix: start Helix with HELIX_LOG=lsp=debug hx myfile.lace and check the log at ~/.cache/helix/helix.log.

Zed: open the debug panel via Help → Open Log File and filter for lace.

⚡ Still stuck?

Open an issue on the GitHub issue tracker and include:

  • Output of lace --version
  • Your editor name and version
  • The relevant section of your LSP config
  • Any error messages from the LSP log
